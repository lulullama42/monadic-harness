## 1. Runtime Progress Feedback

After each driver cycle (decide -> dispatch -> observe), the main agent
displays a progress line to the user. This is the primary feedback mechanism
during task execution.

### Format Specification

```
[Step N/M] phase:task_id status | conditions
  -> signal line (one-line observation summary)
  -> next: hint
```

Field breakdown:

| Field | Meaning |
|-------|---------|
| `[Step N/M]` | Current step number / total fuel budget |
| `phase:task_id` | Current meta phase and task node being executed |
| `âś“` or `âś—` | Success or failure of the dispatched step |
| Conditions | Key condition values from the observation (e.g. `completeness=full`) |
| Signal line | One-line narrative extracted from the observation |
| Next hint | What the driver will do next, based on condition matching |

### Sequential Execution

A single task node completing:

```
[Step 3/30] exec:t2 âś“ completeness=full | surprise=0.2
  â†’ created vite.config.ts, resolved alias mapping
  â†’ next: [t3a, t3b, t3c] (parallel)
```

### Parallel Dispatch

When the task graph specifies parallel nodes, the progress line adapts to
show group status. Two lines are emitted: one at dispatch time and one at
completion.

Dispatch:

```
[Step 4/30] exec:[t3a,t3b,t3c] (parallel)
  â†’ spawning 3 subagents...
```

Completion:

```
[Step 4/30] exec:[t3a,t3b,t3c] âś“âś“âś— | surprise=0.6
  â†’ t3a: migrated app entry âś“
  â†’ t3b: migrated admin entry âś“
  â†’ t3c: dynamic require not supported âś—
  â†’ next: retry t3c
```

Each parallel node gets its own status character (`âś“` or `âś—`), and each gets
a signal sub-line. The group consumes a single step of fuel regardless of
how many nodes run in parallel (see [[04-python-driver]]).

### Phase Transitions

Meta-level transitions (plan, exec, verify, replan) are displayed with the
`[meta]` prefix instead of a step number:

```
[meta] verify â†’ goal alignment check
[meta] verify âś— â†’ replan (attempt 2)
```

These do not consume fuel. They indicate that the driver is transitioning
between meta phases â€” see [[02-control-flow]] for the meta state machine.

### Verbosity Levels

The `progress_verbosity` setting in `config.yaml` controls how much detail
is shown:

| Level | Behavior |
|-------|----------|
| `minimal` | Step line only, no signal or next hint |
| `normal` | Full format as described above (default) |
| `verbose` | Full format plus raw condition values from observation |

### Stderr Warnings

The `observe` command may emit warnings to stderr (per [[specs]] I6):

- **Fuel warning**: `WARNING: Low fuel: {N} steps remaining` â€” when `fuel_remaining <= fuel_warning_threshold` (default 5) and fuel > 0. The main agent should relay this to the user.
- **Invariant warning**: `INVARIANT:{reason}` â€” when an invariant fires (loop detection, drift, fuel exhaustion). This signals that the next `decide` call will process a forced transition. The main agent does not need to act on this directly â€” `decide` handles it.

## 2. Task Summary Report

Auto-generated when a task completes (status `done`) or is aborted (status
`aborted`). Written to:

```
~/.mh/tasks/{task-id}/dataflow/artifacts/task-report.md
```

This report is the primary artifact for post-task review. It is generated by
the driver from structured data â€” no LLM tokens are spent producing it.

### Full Example

```markdown
# Task Report: migrate-webpack-to-vite

## Goal
Migrate monorepo from webpack 5 to vite

## Result
**Status**: Done
**Steps**: 8/30 used
**Duration**: 8m 23s
**Replan count**: 1

## Execution Timeline
| Step | Task | Result | Surprise |
|------|------|--------|----------|
| 1 | t1 | âś“ full | 0.1 |
| 2 | t2 | âś“ full | 0.3 |
| 3 | t3a_t3b_t3c | âś“ full | 0.6 |
| 4 | t3c | âś“ full | 0.2 |
| 5 | t4 | âś“ full | 0.0 |
| 6 | t5 | âś— partial | 0.7 |
| - | resolve: replan | - | - |
| - | compile plan | - | - |
| 7 | t6 | âś“ full | 0.2 |
| 8 | t7 | âś“ full | 0.0 |

## Profile (Accumulated Knowledge)
- build_system: webpack 5 â†’ vite 6.2
- framework: React 18
- entry_points: 3 (app, admin, worker)
- special: uses webpack-dev-middleware in Express server
- special: worker entry uses dynamic require â†’ needs commonjs plugin

## Artifacts
- `vite.config.ts` â€” new vite configuration
- `package.json` â€” updated scripts and dependencies

## Key Decisions
- Step 6: 3 tests fail (surprise=0.7)
- Step 6: escalation â€” drift detected
- Step 6: resolved â€” replan (tests failing, need compatibility fix)
```

### Content Sources

Each section of the report is assembled from a specific on-disk source:

| Section | Source | Notes |
|---------|--------|-------|
| Goal | `meta.json` | The `goal` field set at init time |
| Result | `state.json` + `meta.json` | Step count, status, duration from timestamps |
| Execution Timeline | `trace/trace.jsonl` | One row per trace entry; parallel groups shown as merged single-step result (per [[specs]] R2) |
| Profile | `profile.json` | Accumulated environment knowledge, dumped as-is |
| Artifacts | `dataflow/artifacts/` | Directory scan, excluding `task-report.md` itself |
| Key Decisions | `trace/trace.jsonl` | Entries where `surprise > 0.5` or action is `replan`/`escalate` |

The Key Decisions section extracts the narrative from the trace entry's
`observation_summary` field (which corresponds to the observation's `signal`
field) for each high-surprise event, providing a human-readable summary of
where the task deviated from the plan and how it recovered. The surprise
threshold is hardcoded at 0.5 (per [[specs]] R4). Status is displayed
as a simple capitalized string for v1; `Done (verified)` will be added when
meta-verify is implemented.

See [[03-data-flow]] for the full schema of each source file.

## 3. Mid-Task Status

The user can request the current state of a running task at any time. The
main agent routes this to the driver's `status` command.

### Invocation

```
pymh status
```

### Output Format

```
Task: migrate-webpack-to-vite
Phase: exec | Step: 5/30 | Fuel: 25
Current: t3c (attempt 2) â€” migrate worker entry point
Profile: {build: webpack5â†’vite6.2, entries: 3, done: 2/3}
Last: âś— dynamic require not supported (surprise=0.6)
Next: retry t3c or escalate
```

Field breakdown:

| Field | Source |
|-------|--------|
| Task | `meta.json` â†’ `id` |
| Phase / Step / Fuel | `phase.json` â†’ `phase`, `state.json` â†’ `step`, `fuel` |
| Current | `cursor.json` â†’ `current_task`, `task_attempts` + task graph description |
| Profile | `profile.json` â†’ compact summary of key-value pairs |
| Last | Most recent `trace.jsonl` entry â†’ result + surprise |
| Next | Driver condition evaluation â†’ predicted next transition |

The status output is generated entirely by the driver at zero token cost.
The main agent reads the stdout and relays it to the user.

## 4. User Commands

Available during task execution. These are not formal CLI commands â€” the user
speaks naturally to the main agent, which recognizes intent and routes to the
appropriate action.

| Command | Action | Driver Call |
|---------|--------|------------|
| "status" | Show current task state | `pymh status` |
| "abort" | Stop task, generate report | `pymh abort` |
| "add fuel N" | Extend step budget by N | `pymh fuel --add N` |
| "show profile" | Display accumulated knowledge | Read `profile.json` directly |
| "show trace" | Display execution history | Read `trace/trace.jsonl` directly |
| "show plan" | Display current task graph | Read `ctrlflow/task-graph.yaml` directly |

### Routing

The main agent is the user-facing interface. It holds the orchestration
protocol (see [[05-skill-protocol]]) which includes instructions for
recognizing these commands. When the user says something like "how's it
going?" or "what step are we on?", the main agent maps that to `status`.

Commands that call the driver (`status`, `abort`, `fuel`) go through
`pymh` and return structured output. Commands that read files directly
(`show profile`, `show trace`, `show plan`) have the main agent read and
summarize the file content.

### Abort Behavior

When the user says "abort" or "stop":

1. The driver writes `status: aborted` to `meta.json`.
2. The driver generates the task summary report (same format as completion).
3. The main agent confirms to the user and displays the report summary.

No partial work is lost â€” all observations, trace entries, and artifacts
written before the abort remain on disk.

### Fuel Management

When the user says "add fuel 10" or "give me 10 more steps":

1. The main agent calls `pymh fuel --add 10`.
2. The driver increments `fuel` in `state.json` and updates the total.
3. The main agent confirms: "Fuel extended to 35 (was 25, added 10)."

The driver also emits a fuel warning when remaining fuel drops to the
`fuel_warning_threshold` (default 5). The main agent relays this as:

```
[fuel] 5 steps remaining â€” say "add fuel N" to extend
```

## 5. Templates

Location: `~/.mh/templates/`

Templates provide starting-point task graphs that the plan subagent can use
as a reference when generating a plan. They encode common task patterns so
that the plan subagent does not have to invent structure from scratch.

### Shipped Presets

Installed with `pymh`:

| Template | Pattern | Description |
|----------|---------|-------------|
| `general.yaml` | Generic | Default template. Minimal structure: search/analyze -> execute steps -> verify. |
| `migration.yaml` | Codebase migration | search guides -> plan migration -> migrate entries in parallel -> verify build & tests. |
| `research.yaml` | Research & analysis | search sources -> evaluate quality -> synthesize findings -> verify completeness. |

### User Custom Templates

Users can add their own `.yaml` files to `~/.mh/templates/`. Any file placed
there becomes available for selection. No registration step is needed â€” the
plan subagent's instruction lists available templates by scanning the
directory.

### Template Format

Templates follow the NL plan format specified in [[05-skill-protocol]]: a
YAML file with a `plan.steps[]` structure. Each step defines a task node
with its description, conditions, and transitions. The plan subagent treats
template content as a starting point, not a rigid constraint â€” it may add,
remove, or modify steps based on the specific goal.

### Selection

Template selection happens at task init time:

1. **Explicit**: the user specifies `--template migration` when initializing.
   The plan subagent receives the template content as reference material.
2. **Default**: if no template is specified, `general.yaml` is used (or
   whatever `defaults.template` is set to in `config.yaml`).
3. **From scratch**: the plan subagent may choose to ignore the template
   entirely and plan from scratch if the goal does not fit any available
   pattern.

## 6. Configuration

Global configuration lives at `~/.mh/config.yaml`. Created on first `pymh`
install with sensible defaults. Users can edit it directly with any text
editor.

### Full Schema

```yaml
# Default settings
defaults:
  fuel: 30                    # default fuel for new tasks
  template: general           # default template name
  max_task_attempts: 3        # loop detection threshold
  max_consecutive_failures: 3 # drift check threshold
  drift_threshold: 2.0        # surprise accumulation limit
  fuel_warning_threshold: 5   # fuel low warning
  max_replan_count: 3         # max replans before abort

# User preferences
preferences:
  progress_verbosity: normal  # minimal | normal | verbose
  report_format: markdown     # markdown (only option for v1)
```

### Settings Reference

| Setting | Type | Default | Description |
|---------|------|---------|-------------|
| `defaults.fuel` | int | 30 | Initial fuel budget for new tasks |
| `defaults.template` | string | `general` | Template name used when none is specified |
| `defaults.max_task_attempts` | int | 3 | Max retry attempts per task node before loop detection triggers |
| `defaults.max_consecutive_failures` | int | 3 | Max consecutive failures before drift check triggers |
| `defaults.drift_threshold` | float | 2.0 | Accumulated surprise limit before escalation |
| `defaults.fuel_warning_threshold` | int | 5 | Remaining fuel level that triggers a warning to the user |
| `defaults.max_replan_count` | int | 3 | Max replans before forcing abort |
| `preferences.progress_verbosity` | enum | `normal` | Detail level of progress lines: `minimal`, `normal`, `verbose` |
| `preferences.report_format` | enum | `markdown` | Output format for task reports (only `markdown` in v1) |

### Override Precedence

1. CLI arguments (e.g. `--fuel 50`) override config values for that task.
2. `config.yaml` values override built-in defaults.
3. Built-in defaults are used when neither CLI nor config specifies a value.
